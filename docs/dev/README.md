# 开发问题与修复记录

本目录用于沉淀 **问题修复类** 变更，便于回溯、评审与 onboarding。

## 何时写

- 修复 Bug、异常行为、接口/配置错误
- 排查并解决性能、兼容性、逻辑错误等问题

纯功能新增、与问题无关的重构可不写；若重构是为修复隐患，应在复盘中说明。

## 文件命名

```
YYYY-MM-DD-<简短描述 slug>.md
```

同一次修复如需多轮提交，可继续编辑同一文件，避免碎片文档。

## 文档模板

复制以下内容到新文件后按需填写。**不必**把「修改点」写成函数/字段清单；以**行为、取舍、边界**为主，路径只在需要对照代码时简略列出（详见 `.cursor/rules/dev-delivery-zh.mdc` → docs/dev 文档写法）。

```markdown
# <标题：一句话概括修复或交付内容>

**日期：** YYYY-MM-DD

## 背景与原因

- 现象 / 需求来源：
- 根因或设计取舍（与旧方案差异可写在这里）：

## 修改点

- 用户或系统侧可见的变化（分条，少用实现细节）：
- （可选）涉及模块：`主要文件或目录`，仅作索引

## 复盘

- 可改进点、风险、未完项：
- 后续建议（可选）：
```

## Git 提交

修复完成后应 **提交到 Git**，且 **commit message 写清楚**（勿只用「fix」或「update」）。

建议格式：

```
<类型>(<范围>): <一句话摘要>

- 原因：…
- 修改：…
- 记录：docs/dev/<文件名>.md
```

`docs/dev` 记录与代码改动宜放在 **同一 commit**。

## 与 Cursor 规则

- **交付与留档（合并规则）：** `.cursor/rules/dev-delivery-zh.mdc` — **原因 / 修改 / 验证 / 记录** 四块在聊天、`docs/dev` 与 `commit message` 之间对齐；内含 **`docs/dev` 文档写法**（设计/行为优先，避免实现清单式堆砌）。
