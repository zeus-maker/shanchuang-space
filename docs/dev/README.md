# 开发问题与修复记录

本目录用于沉淀 **问题修复类** 变更，便于回溯、评审与 onboarding。

## 何时写

- 修复 Bug、异常行为、接口/配置错误
- 排查并解决性能、兼容性、逻辑错误等问题

纯功能新增、与问题无关的重构可不写；若重构是为修复隐患，应在复盘中说明。

## 文件命名

```
YYYY-MM-DD-<简短描述 slug>.md
```

同一次修复如需多轮提交，可继续编辑同一文件，避免碎片文档。

## 文档模板

复制以下内容到新文件后按需填写：

```markdown
# <标题：一句话概括修复内容>

**日期：** YYYY-MM-DD

## 背景与原因

- 现象：
- 根因 / 触发条件：

## 修改点

- `path/to/file` — …
- …

## 复盘

- 可改进点：
- 风险与注意事项：
- 后续建议（可选）：
```

## Git 提交

修复完成后应 **提交到 Git**，且 **commit message 写清楚**（勿只用「fix」或「update」）。

建议格式：

```
<类型>(<范围>): <一句话摘要>

- 原因：…
- 修改：…
- 记录：docs/dev/<文件名>.md
```

`docs/dev` 记录与代码改动宜放在 **同一 commit**。

## 与 Cursor 规则

- **交付与留档（合并规则）：** `.cursor/rules/dev-delivery-zh.mdc` — 含：向用户回复的格式与质量（简体中文）、`docs/dev` 留档与 `git commit` 规范（修复类；功能迭代如需留档亦可按同结构书写）。
